<objective>
Define the Dispatcher interface and build the WhatsApp Web adapter that replaces the Worker's stub dispatch. This establishes the channel adapter architecture that Phase 5 extends with WABA and Telegram.

Purpose: Creates the plugin boundary between the queue worker and channel implementations. After this plan, messages dispatched from JetStream flow through a real whatsmeow client (when a session is connected) instead of being logged and acked.

Output: Dispatcher interface, Registry, WhatsAppAdapter with staggered dispatch, updated Worker that uses the registry.
</objective>

<task type="auto" tdd="true">
  <name>Task 3: WhatsAppAdapter implementing Dispatcher (RED + GREEN)</name>

  <description>
  Build the WhatsApp Web dispatcher adapter:
  - WhatsAppAdapter implements channel.Dispatcher
  - Dispatch(ctx, msg) sends a text message via whatsmeow
  - Staggered dispatch: 1-3s random delay (time.Sleep) before each send
  - Parse recipient JID from msg.To (phone number → @s.whatsapp.net)
  - On send error: check if terminal (403, LoggedOut) → return TerminalError
  - On whatsmeow send success: return nil
  - NewWhatsAppAdapter(whatsappClient *WhatsAppClient) constructor
  </description>

  <deviations>
  - Stagger uses crypto/rand for random delay (not math/rand — better distribution)
  - Error classification: whatsmeow errors that indicate logged-out/forbidden return TerminalError
  - Adapter is stateless beyond the client reference — all state lives in the session
  </deviations>
