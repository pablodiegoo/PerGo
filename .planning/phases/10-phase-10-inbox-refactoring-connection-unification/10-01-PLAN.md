# Plan 10-01: Inbox Polling Stability & Zero-JS Scroll Refactoring

Refactor the conversational inbox's message polling loop to resolve infinite reloading loops, duplicate client-side event listeners, and scroll jitter. This is achieved by utilizing server-side cursors swapped out-of-band (OOB) and relying on native HTMX scroll behavior without custom JavaScript event handlers.

### Manual Verification
1. Run the local PerGo environment.
2. Open the Inbox interface.
3. Select an active conversation.
4. Verify that:
   - Polling requests are dispatched every 3s to `/admin/inbox/messages?after_id=<uuid>`.
   - Incoming messages append to the bottom of the scroll container automatically.
   - Container scrolls down to show the new message.
   - Inspecting the DOM confirms no duplicate event listeners are registered when switching active conversations.

## Must Haves (Goal-Backward Verification)
- [ ] No client-side JavaScript listens to `htmx:afterSwap` to re-trigger polling or rewrite `hx-get` attributes in the inbox layout.
- [ ] Polling response includes `#chat-poll-anchor` with `hx-swap-oob="true"` and an updated cursor matching the newest message UUID.
- [ ] Polling returns 204 No Content when no new messages are available.
