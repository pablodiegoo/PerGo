# Phase 24.2 Research: Close gap: TYPE-04 — populate ConnectionID, SenderIdentity, and TraceID in TypebotForwarder queue message

## Understanding the Gap
Currently, the Typebot integration forwarder (`internal/integration/typebot/forwarder.go`) receives an `inbound.InboundEvent`, communicates with Typebot, and generates outbound reply messages. When publishing these replies to the `messages.outbound` queue, it creates a `domain.QueueMessage` payload but leaves out critical routing fields (`ConnectionID`, `SenderIdentity`, and `TraceID`).
- **ConnectionID** and **SenderIdentity** are required to route the outbound message correctly through the same connection that received the inbound message.
- **TraceID** is required to correlate the message through the audit logs and dispatch systems.

## Decisions & Implementation Path

1. **ConnectionID Population**: 
   - Assign `outMsg.ConnectionID = event.ConnectionID`.
2. **SenderIdentity Population**: 
   - Assign `outMsg.SenderIdentity = event.To`. (The bot is sending the message *from* the identity that the user sent the message *to*).
3. **TraceID Population**:
   - Generate `traceID := uuid.New().String()` *before* constructing the `QueueMessage`.
   - Assign `outMsg.TraceID = traceID`.
   - Use the exact same `traceID` variable for the `f.publisher.Publish(ctx, "messages.outbound", b, traceID)` call.

## Target Changes
### `internal/integration/typebot/forwarder.go`
Modify the loop in `SyncInboundMessage` starting around line 158:
```go
	// 5. Publish returned messages from Typebot to messages.outbound
	for _, m := range messages {
		traceID := uuid.New().String()
		// Create an outbound QueueMessage
		outMsg := domain.QueueMessage{
			WorkspaceID:    event.WorkspaceID,
			ConnectionID:   event.ConnectionID,
			SenderIdentity: event.To,
			TraceID:        traceID,
			Channel:        event.Channel,
			To:             event.From,
			Body:           extractText(m),
		}

		// Ensure body is not empty before sending
		if outMsg.Body != "" {
			b, err := json.Marshal(outMsg)
			if err != nil {
				slog.Error("failed to marshal outbound message from typebot", "error", err)
				continue
			}

			err = f.publisher.Publish(ctx, "messages.outbound", b, traceID)
			if err != nil {
				slog.Error("failed to publish outbound message from typebot", "error", err)
			}
		}
	}
```

## Validation Architecture

To test the implementation and verify that TYPE-04 is resolved, follow this validation plan:

1. **Unit Test Coverage**:
   - Ensure a unit test for `TypebotForwarder.SyncInboundMessage` asserts that the payload passed to the mock `Publisher` contains the correct `ConnectionID`, `SenderIdentity`, and `TraceID`.
   - Inject a mock publisher and verify the unmarshaled payload properties match the input event's `ConnectionID` and `To` values, and that `TraceID` is correctly populated.

2. **Integration Test (Manual or E2E)**:
   - **Setup**: Start the PerGo application with a Typebot integration configured.
   - **Action**: Simulate an inbound message (via webhook or test UI) that triggers a Typebot bot flow.
   - **Assert**: Check the `messages.outbound` NATS JetStream queue (e.g., using the NATS CLI: `nats stream view messages`) to inspect the raw message payload.
   - **Verify**: The payload JSON must contain:
     - `connection_id` matching the inbound message's connection.
     - `sender_identity` matching the `To` field of the inbound message.
     - `trace_id` matching a valid UUID string, which matches the message header trace.

3. **System Behavior Verification**:
   - Check the outbound dispatcher worker logs. The worker should successfully route the message out using the provided `ConnectionID` and `SenderIdentity`.
   - Verify that there are no routing failures (e.g., "unknown connection" or "sender identity missing") in the logs when Typebot replies.
