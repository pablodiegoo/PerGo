# Phase 24.2.1: fix-typebot-message-construction-gap - Context

**Gathered:** 2026-07-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Implement message construction logic within `TypebotForwarder` to act as a bridge mapping PerGo webhook payloads to the Typebot start-chat API.
</domain>

<decisions>
## Implementation Decisions

### Payload Structure
- **D-01:** Nest PerGo's metadata under a `pergo_metadata` or `custom_data` object to avoid collisions.
- **D-02:** Forward raw text exactly as received to preserve user intent.
- **D-03:** Map PerGo's `SenderIdentity` (or combined) to Typebot's `sessionId` root field.
- **D-04:** Combine ConnectionID and SenderIdentity (e.g., `<ConnectionID>:<SenderIdentity>`) for the Typebot `sessionId` to ensure uniqueness per channel.

### Media & Attachments
- **D-05:** Pass the public URL of the media (if available/uploaded by PerGo) for media attachments.
- **D-06:** Send the media URL as a separate metadata field inside `pergo_metadata.media_url` and use a placeholder for `message`.
- **D-07:** Use `[Media Attachment]` as the placeholder text in the `message` field.
- **D-08:** Include the media type (e.g., `image`, `video`, `audio`) explicitly in the `pergo_metadata` payload.
- **D-09:** For unsupported message types (e.g., location, contact), send a placeholder text (e.g., `[Unsupported message: location]`) in the `message` field to hand off control to Typebot's native fallback/timeout mechanisms.

</decisions>

### Established Patterns

<specifics>
## Specific Ideas

The core point is that PerGo is a bridge between the channel and Typebot on this case. Unsupported messages should be passed with placeholders so Typebot can handle fallbacks itself.
