# Plan: 21-02 — Bidirectional Sync Engine & Message Dispatch

This plan covers Wave 2 of the Chatwoot Integration. It completes the outbound message ingestion pipeline from Chatwoot to PerGo's queue and implements the inbound customer message synchronization to Chatwoot.

  <action>
    Create internal/integration/chatwoot/client.go implementing ChatwootClient wrapping Chatwoot REST v1 endpoints:
    - SearchContact(ctx, identifier): GET /api/v1/accounts/{account_id}/contacts/search?q={identifier}.
    - CreateContact(ctx, identifier, name, phoneNumber, customAttributes): POST /api/v1/accounts/{account_id}/contacts.
    - UpdateContact(ctx, chatwootContactID, name, phoneNumber, customAttributes): PUT /api/v1/accounts/{account_id}/contacts/{contact_id}.
    - CreateConversation(ctx, contactID, inboxID): POST /api/v1/accounts/{account_id}/conversations.
    - PostMessage(ctx, conversationID, content, private): POST /api/v1/accounts/{account_id}/conversations/{conversation_id}/messages.
    All calls must handle 4xx client errors as terminal (returning a custom error like ErrNotFound for HTTP 404 response) and 5xx/timeouts as retriable.

    Create internal/integration/chatwoot/syncer.go implementing the ChatwootSyncer interface (which is declared in the internal/inbound package to avoid circular Go dependencies):
    - SyncInboundMessage(ctx, contact *domain.Contact, ev *InboundEvent): checks mappingRepo cache first to prevent remote search API calls (per D-04) for (ev.WorkspaceID, contact.ID, ev.ConnectionID). If mapped, posts content to PostMessage. If the PostMessage call fails with a 404 Not Found error (indicating the conversation or contact does not exist in Chatwoot), delete the local mapping row using mappingRepo.Delete(ctx, ev.WorkspaceID, contact.ID, ev.ConnectionID), and proceed to re-resolve the contact and re-create the conversation. If not mapped from the start, queries SearchContact using contact ID (UUID string) as query 'identifier'. If contact is found, calls UpdateContact to synchronize current attributes (e.g. name, phone number, custom attributes). If not found, calls CreateContact. After resolving the contact, calls CreateConversation, calls MappingRepo.Upsert (storing connection_id and sender_identity along with IDs and channel), and finally calls PostMessage.
    
    Implement unit/integration tests with httptest.NewServer mock endpoints in client_test.go and syncer_test.go.
  </action>
  <verify>
    Run syncer and client unit tests:
    `go test -run TestChatwootClient ./internal/integration/chatwoot/...`
    `go test -run TestChatwootSyncer ./internal/integration/chatwoot/...`
  </verify>
  <acceptance_criteria>
    - ChatwootClient wraps SearchContact, CreateContact, UpdateContact, CreateConversation, and PostMessage calls using the account details and Access Token.
    - CreateContact and UpdateContact support phone_number and customAttributes parameters.
    - ChatwootSyncer checks local mapping cache using workspace_id, contact_id, and connection_id to prevent cross-talk.
    - SyncInboundMessage updates the Chatwoot contact details using UpdateContact if the contact exists, and creates it if not found.
    - All mock server client/syncer tests pass.
  </acceptance_criteria>
</task>

  <action>
    Fully implement the ChatwootWebhookHandler in internal/api/handler/chatwoot_webhook.go.
    In the Handle method:
    1. Parse incoming webhook payload ChatwootWebhookPayload.
    2. Check filters: message_type must be "outgoing", private must be false, and sender.type must be "user" (human agent reply) (per D-06). If any filter condition is not met, return 200 OK and return early.
    3. Retrieve workspace ID from request context (set by AuthMiddleware).
    4. Query ChatwootMappingRepository using both the retrieved workspace ID and the Chatwoot Conversation ID to find the mapped mapping (which includes workspace_id, contact_id, channel, connection_id, and sender_identity). This ensures strict multi-tenancy isolation.
    5. Query contact_identities using the mapped contact_id, channel, and workspace_id to resolve the customer's channel identity (sender_identity column in contact_identities) which represents the destination address.
    6. Construct domain.QueueMessage setting WorkspaceID, ConnectionID (from mapping), SenderIdentity (from mapping, representing our bot/phone), To (the resolved customer's channel identity), Channel (from mapping), and Body (webhook content).
    7. Publish the QueueMessage payload to NATS JetStream subject "messages.outbound" to dispatch the message.

    Update cmd/pergo/main.go to register/wire the real ChatwootWebhookHandler dependencies (ChatwootMappingRepository, contact identities database access/queries, and the NATS publisher) and mount the handler to the webhook route in the Echo engine.

    Write integration tests in chatwoot_webhook_test.go confirming that webhook handles filtering, customer identity resolution, and queue publishing with correct routing parameters.
  </action>
  <verify>
    Run webhook handler tests:
    `go test -run TestChatwootWebhookHandler ./internal/api/handler/...`
  </verify>
  <acceptance_criteria>
    - Webhook rejects private notes, user inbound payloads, or bot messages by returning HTTP 200 OK immediately without queue publishing.
    - Webhook maps Conversation ID to mapped contact, connection_id, and sender_identity (our outgoing channel identity) from the mapping table.
    - Webhook resolves customer's channel identity (to field) via contact_identities database query.
    - Webhook publishes domain.QueueMessage to JetStream subject 'messages.outbound' with ConnectionID, SenderIdentity, and To correctly set.
    - cmd/pergo/main.go successfully wires ChatwootWebhookHandler with repositories and NATS publisher, and mounts the endpoint.
    - Integration tests cover event filtering, identity resolution, and NATS messaging queue output.
  </acceptance_criteria>
