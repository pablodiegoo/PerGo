# Plan 1: Schema, Inbound "to" Enrichment, and Repository Queries

This plan implements the database optimizations, enriches inbound event payloads with recipient identities, updates the recipient sessions schema for multi-instance isolation, and builds the repository query layer for the split-pane inbox.

## Tasks

<task id="migration_recipient_sessions_to_column">
  <read_first>
    - internal/platform/postgres/migrations/012_consolidate_connections.sql
    - internal/platform/postgres/migrations/006_create_recipient_sessions.sql
  </read_first>
  <action>
    Create a new goose migration file at `internal/platform/postgres/migrations/013_recipient_sessions_to_column.sql` to add the `recipient_identity` column, update the primary key of `recipient_sessions`, and add a compound performance index on `audit_logs`:
    ```sql
    -- +goose Up
    ALTER TABLE recipient_sessions ADD COLUMN recipient_identity TEXT NOT NULL DEFAULT '';
    ALTER TABLE recipient_sessions DROP CONSTRAINT recipient_sessions_pkey;
    ALTER TABLE recipient_sessions ADD PRIMARY KEY (workspace_id, recipient_phone, channel, recipient_identity);

    CREATE INDEX IF NOT EXISTS idx_audit_logs_inbound_grouping
      ON audit_logs (workspace_id, event_type, (payload->>'from'), (payload->>'channel'), (payload->>'to'), created_at DESC);
    
    -- +goose Down
    DROP INDEX IF EXISTS idx_audit_logs_inbound_grouping;
    ALTER TABLE recipient_sessions DROP CONSTRAINT recipient_sessions_pkey;
    ALTER TABLE recipient_sessions ADD PRIMARY KEY (workspace_id, recipient_phone, channel);
    ALTER TABLE recipient_sessions DROP COLUMN IF EXISTS recipient_identity;
    ```
  </action>
  <acceptance_criteria>
    - The new migration file is created under the correct path.
    - Running database migrations via `make migrate` or equivalent script succeeds.
    - Schema changes and indexes are verified in PostgreSQL.
  </acceptance_criteria>
</task>

Ensure the UNION query matches specifically on the recipient identity to prevent multi-instance leaks:
