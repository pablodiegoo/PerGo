# Plan 3: Interactive Chat Panel, Send Replies, and Polling

This plan implements the interactive message thread view. It builds the chat panel, alternating bubbles, auto-growing input textarea, message polling (3s) using a UUID cursor, send handler mapping to NATS JetStream, and background unread alerts via in-page toasts.

## Tasks

<task id="create_chat_panel_templates">
  <read_first>
    - templates/pages/inbox.templ
  </read_first>
  <action>
    Create active chat template components:
    1. `templates/components/chat_panel.templ`: structures the chat view header (contact and channel details), the message history viewport, and the send textarea. The message history viewport must poll `/admin/inbox/messages` every 3s, appending new messages to the end of the viewport (`hx-swap="beforeend"`), passing the last message ID as an `after` cursor. The textarea must handle auto-resize and submit on Enter (excluding Shift+Enter).
    2. `templates/components/message_bubble.templ`: renders single bubbles. Left-aligned white cards for inbound messages (with contact initials avatar); right-aligned blue (`#3b82f6`) cards for outbound replies (with checkmarks).
    3. `templates/components/inbox_toast.templ`: lightweight toast layout positioned top-center, which dismisses itself automatically after 3.5 seconds.
  </action>
  <acceptance_criteria>
    - Templates compile using `templ generate`.
    - Auto-grow JavaScript triggers correctly as operator types.
    - Submit triggers HTMX POST on Enter keypress.
  </acceptance_criteria>
</task>

    - Updates the server-side last read timestamp for this recipient session by calling `RecipientSessionRepository.UpdateLastReadAt` with the current timestamp.

  </action>

<task id="implement_live_polling_handler">
  <read_first>
    - internal/api/handler/admin/inbox.go
    - internal/repository/recipient_session.go
  </read_first>
  <action>
    Implement `PollMessages` in `internal/api/handler/admin/inbox.go`:
    - Accepts `contact`, `channel`, `recipient_identity`, and the `after` UUID cursor.
    - Queries new messages using `AuditRepository.ListThread` passing the `after` cursor.
    - If new messages are for the currently open conversation, renders the message bubbles and updates the server-side last read timestamp by calling `RecipientSessionRepository.UpdateLastReadAt`.
    - If new messages belong to a different conversation, set the `HX-Trigger` header in the response: `{"showToast":{"text":"Nova mensagem de " + contact}}`.
    - Return 200 OK (empty body if no new messages are rendered).
  </action>
  <acceptance_criteria>
    - Polling returns only the new message bubbles since the `after` UUID.
    - Polling for a different contact triggers the `showToast` event via `HX-Trigger` response headers.
  </acceptance_criteria>
</task>

<task id="implement_send_message_handler">
  <read_first>
    - internal/api/handler/admin/inbox.go
    - cmd/pergo/main.go
    - internal/repository/connection.go
  </read_first>
  <action>
    Inject `ConnectionRepository` and `RecipientSessionRepository` into `InboxHandler` (in `main.go` and `inbox.go`).
    Implement `SendMessage` in `internal/api/handler/admin/inbox.go`:
    - Binds form parameters: `contact` (maps to `to`), `channel`, `recipient_identity` (maps to `sender_identity`), and `body`.
    - Resolves active `workspace_id`.
    - Validates inputs (non-empty body).
    - Looks up the connection by `sender_identity` (using `ConnectionRepository.GetBySenderIdentity`) using the active `workspace_id` and the `recipient_identity`.
    - Constructs and publishes a new `domain.QueueMessage` to NATS JetStream (subject: `messages.outbound`), populating `ConnectionID` with the resolved connection's ID.
    - Returns a blank response with `200 OK` or `204 No Content` (so HTMX clears the input or re-polls naturally).
  </action>
  <acceptance_criteria>
    - POST `/admin/inbox/send` publishes to the JetStream broker.
    - Empty messages return HTTP 400.
    - Replied messages are processed by NATS and dispatched.
    - NATS QueueMessage payload contains the correct resolved connection ID.
  </acceptance_criteria>
</task>
