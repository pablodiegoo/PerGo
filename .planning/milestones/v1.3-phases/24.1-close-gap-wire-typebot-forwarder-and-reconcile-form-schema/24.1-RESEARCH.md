**Researched:** 2026-07-19
**Domain:** Composition-root wiring + form/config schema reconciliation + integration test gap
**Confidence:** HIGH (all four critical facts verified directly against the codebase at the cited line numbers)

## Summary

Phase 24.1 is a pure plumbing + schema reconciliation phase, not new logic. The substantive work was already done in Phase 22: `typebot.Forwarder` and `typebot.Client` are implemented and unit-tested, and `TypebotWebhookHandler` is implemented and routed. What is broken is the *composition root* — `cmd/pergo/main.go:215` constructs the inbound router with `nil` for the Typebot forwarder, and the admin form (`templates/pages/typebot_settings.templ`) produces a config that the forwarder cannot consume (per-bot `public_token`, `trigger_words []string`, `connection_id`, `is_default` are missing; a top-level `public_token` and single `trigger_keyword` are written instead).

The fix has four well-scoped parts:

1. **Wire `typebot.Forwarder` in `main.go`** (mirror the `chatwoot.NewChatwootSyncer(...)` pattern at line 214, pass to `inbound.NewDefaultRouter(chatwootSyncer, typebotForwarder)` at line 215). Requires instantiating `*repository.TypebotSessionRepository(pool)` and importing the `internal/integration/typebot` package (currently zero references in main.go).
2. **Reconcile form schema with `typebot.Config`** — change `pages.TypebotConfig` and `templates/pages/typebot_settings.templ` to produce the per-bot shape that `typebot.BotConfig` expects, including a connection picker. The handler `admin/typebot.go` already calls `c.FormValue` for the existing four fields; it must be extended to read per-bot fields and a connection ID.
3. **Add happy-path tests for `TypebotWebhookHandler`** — current `typebot_webhook_test.go` only asserts 401 on missing workspace context. The chatwoot equivalent at `chatwoot_webhook_test.go:151-348` is the pattern to follow (uses `fakePublisher`, `getTestPool`, real workspace + connection + contact, posts a valid payload, asserts on `pub.published`).
4. **Add a composition-root smoke test** — the audit's tech-debt section explicitly recommends a test that asserts `chatwootSyncer != nil && typebotForwarder != nil` after `inbound.NewDefaultRouter(...)` to prevent the same regression class from recurring.

**Primary recommendation:** Three discrete tasks — (a) wire forwarder + reconcile form schema, (b) add happy-path webhook test, (c) add composition-root smoke test — committed atomically with a single full test run. No migrations are needed; no new packages are introduced. The form schema change is the only place where backward-compat with already-stored Typebot configs is at risk (see Concerns §4).

### Pattern 2: Form Schema Reconciliation

**What:** Change the form so that, when submitted, the handler builds a JSON byte array that round-trips cleanly through `typebot.Config` and `typebot.BotConfig`. The current form does not.

**Schema delta:**

| Field | Current (`pages.TypebotConfig`) | Target (`typebot.Config`) |
|-------|----------------------------------|---------------------------|
| Top-level `api_url` | `APIURL string` | `APIURL string` ✓ (matches) |
| Top-level `public_token` | `PublicToken string` (written) | **Not used** — must be removed (or demoted to legacy fallback; see Concerns §4) |
| Per-bot `bot_id` | `BotID string` | `BotID string` ✓ |
| Per-bot `public_token` | **missing** | `PublicToken string` (required) |
| Per-bot `connection_id` | **missing** | `ConnectionID string` (required by `forwarder.go:68, 102` — the forwarder matches `b.ConnectionID == event.ConnectionID.String()`) |
| Per-bot `trigger_words` | `TriggerKeyword string` (single) | `TriggerWords []string` (array) |
| Per-bot `is_default` | **missing** | `IsDefault bool` (drives the fallback in `forwarder.go:69-71`) |

**Form approach (minimal-change):**
- Keep the existing API URL input.
- Replace the top-level "Public Token" input with a **per-bot** "Bot Public Token" input inside the bot settings card.
- Add a **Connection** `<select>` populated by `repository.ConnectionRepository.List(workspaceID, ...)` so the operator picks a known connection rather than typing a UUID. (Alternative: keep a text input and validate the UUID; see Discretion.)
- Replace the single "Trigger Keyword" input with either a comma-separated text input parsed into `[]string`, or — for the operator console — a single keyword persisted as `[]string{keyword}`. **The former matches the forwarder's array contract and avoids a second schema break later.**
- Add a hidden `<input type="checkbox" name="is_default" value="true">` so the operator can flag the default bot per connection.

**Handler change (`admin/typebot.go`):**
- Read `c.FormValue("bot_id")`, `c.FormValue("bot_public_token")`, `c.FormValue("connection_id")`, `c.FormValue("trigger_keywords")` (comma-split), `c.FormValue("is_default") == "true"`.
- Build `typebot.Config{APIURL: apiURL, Bots: []typebot.BotConfig{{BotID: botID, PublicToken: botPublicToken, ConnectionID: connectionID, TriggerWords: splitKeywords, IsDefault: isDefault}}}` directly. **This removes the indirection through `pages.TypebotConfig` for the storage path.** The `pages.TypebotConfig` shape remains in the templ file as a *display* model only, populated when reading the stored JSON for the GET form.
- Validate required fields with clear error messages; on parse failure re-render the form with `errorMsg` populated (mirrors the existing `if apiURL == "" || botID == ""` branch at line 82).

### Pattern 4: Composition-Root Smoke Test

**What:** A test that constructs the inbound router the same way `main.go` does, and asserts both forwarders are non-nil. This test will fail if either forwarder is dropped from the wiring (the same regression class that caused this phase).

**Test outline:**
- File: `cmd/pergo/main_test.go` (new) or `cmd/pergo/wiring_test.go` (new).
- Build a `*pgxpool.Pool` and a real NATS connection (or use the existing `getTestPool` helper if exported; otherwise inline a minimal pool/NATS for the smoke test).
- Replicate lines 212-215: build repos, build both forwarders, call `inbound.NewDefaultRouter`.
- Assert `chatwootSyncer != nil` and `typebotForwarder != nil` via reflection on the unexported router fields, or expose accessors. The cleanest approach: change `internal/inbound/router.go` to add a tiny accessor `(r *DefaultInboundRouter) ChatwootSyncer() ChatwootSyncer` and `TypebotForwarder() TypebotForwarder`, used only in this test. **However**, that adds API surface to production code purely for a test — a less invasive alternative is a `_test.go` file in package `inbound` (white-box) that pokes the unexported fields directly. Recommended: add a `_test.go` in `cmd/pergo` that does reflection with a clear comment explaining the intent.
- Recommend the **white-box** approach: put the test in `package main` (since `cmd/pergo/main.go` is `package main`) and reference the local variables directly. This requires moving the wiring into a function `buildInboundRouter(...) (chatwootSyncer, typebotForwarder, router, err)` so it can be called from tests. See the planner discretion section for the trade-off.

**Precedent in the codebase:** No prior composition-root smoke test exists (this is a recommended addition per the audit's tech-debt section). The closest analog is `internal/inbound/router_test.go:39-105` which tests the router itself in isolation, but that test injects mocks — it does not catch a wiring mistake in `main.go`.

### Anti-Patterns to Avoid

- **Anti-pattern: Hand-rolling a `typebot.Config` JSON unmarshaler that forgives the old form shape.** The form is the *only* writer of this JSON in production; if the form produces the right shape, the JSON round-trips trivially. A custom unmarshaler would add complexity for zero functional benefit and would create a second "shape" of stored config that future readers must handle.
- **Anti-pattern: Keeping the top-level `public_token` as a fallback.** The forwarder never reads a top-level `public_token` (verified: only `BotConfig.PublicToken` is referenced in `forwarder.go` at lines 110, 141). Keeping the field in the form would mislead the operator and the data would be silently ignored.
- **Anti-pattern: Calling `typebot.NewForwarder` with a real `*http.Client` wrapped in retry logic at composition time.** `typebot.NewForwarder` (line 24-35) already creates a `NewClient()` (line 64: `&http.Client{}`) internally. The forwarder is invoked in a goroutine with a 10s timeout (`router.go:42-43`), so resilience is already bounded. Adding retry at the composition layer is over-engineering for this phase.
- **Anti-pattern: Using `string` for `BotConfig.ConnectionID` and parsing on every match.** This is already how the code works (`forwarder.go:68, 102` does `b.ConnectionID == event.ConnectionID.String()`). The schema reconciliation must preserve `ConnectionID string` in `typebot.BotConfig` to avoid touching `forwarder.go`. Trying to "fix" it now expands scope beyond gap-closure.
- **Anti-pattern: Adding a "multi-bot" UI.** The audit and the form file (line 74: `<!-- Simplified to single bot for now since UI doesn't mention multiple complex setup -->`) explicitly accept single-bot. The data model already supports multiple `BotConfig` entries per workspace; the form just persists one. Do not add HTMX "Add Bot" infrastructure in a gap-closure phase.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Connection picker UI | Hand-rolled list of `<input>` for connection IDs | `<select>` populated by `repository.ConnectionRepository.List(workspaceID, ...)` | The connection list is the operator's source of truth; typing a UUID by hand guarantees typos. `ConnectionRepository.List` already exists. |
| Trigger keyword splitting | Custom parser for "comma or space separated" | `strings.Split(c.FormValue("trigger_keywords"), ",")` + `strings.TrimSpace` per element | Typebot's `trigger_words` are exact-match against `bodyLower` (verified at `forwarder.go:72-74`). Simple comma split is sufficient and matches operator mental model. |
| Form binding for dynamic bot count | HTMX "Add Bot" + JSON-array form encoding | Single bot in the form; single `BotConfig` in the stored JSON | The audit's gap-closure intent is "form matches forwarder's expected shape" — single bot achieves that. Multi-bot UI is out of scope. |
| Composition-root smoke test | Mocking `inbound.NewDefaultRouter` to assert call args | Real call to `inbound.NewDefaultRouter` with real (but nil-pointer-safe) deps, then reflection on the unexported fields | The point of the test is to catch "the second arg is `nil`" — only a real call can do that. |

**Key insight:** This phase is a regression fix, not a refactor. The forwarder is already correct. The fix is *not* in the forwarder; the fix is in how the forwarder is constructed and what the form writes. Adding new abstractions (custom unmarshaler, multi-bot UI, retry logic) would expand scope and risk introducing the same class of regression.

## Common Pitfalls

### Pitfall 1: Backward-compat with already-stored Typebot configs

**What goes wrong:** A workspace that saved a Typebot config in Phase 22 has a JSON envelope like:
```json
{"api_url":"https://...","public_token":"OLD_TOKEN","bots":[{"bot_id":"X","trigger_keyword":"sales"}]}
```
The new form will overwrite this when the operator clicks "Save". If the new form lacks a `PublicToken` (top-level) field, the new JSON will not contain it — fine for new writes, but the form's *GET handler* (`admin/typebot.go:46-50`) must still unmarshal whatever is in the DB into the display model. If `pages.TypebotConfig` no longer has a `PublicToken` field, the old field is silently dropped on next save.

**Why it happens:** The audit flagged the form mismatch as TYPE-01 partial. The reconciliation must include either:
- (a) A migration that decodes the old shape and re-saves it in the new shape (impractical without operator action).
- (b) A GET handler that tolerates both shapes (read into a permissive struct, then convert to `typebot.Config` for storage).
- (c) Acceptable data loss: existing Typebot configs are not migrated; operators must re-save through the new form. The audit implies (c) is acceptable because v1.3 has not shipped to production (TYPE-04 is unsatisfied, so no customer is yet in an active Typebot session).

**How to avoid:** Document the data-loss risk in the plan's "Concerns" or "Tech Debt" section. If a single workspace happens to have stored a Phase-22-era Typebot config, that config is effectively stale and the operator must re-enter it. (No operators exist yet — v1.3 is being closed before external release.)

**Warning signs:** A test that pre-populates the `integrations` table with a Phase-22-era config and asserts the GET form renders without error would fail under approach (c). Approach (b) avoids this.

**Recommended resolution:** Approach (b) — make `pages.TypebotConfig` *display-only* and build `typebot.Config` *storage-only*. The GET handler unmarshals into `typebot.Config` (which has no `PublicToken` top-level field, so the old `public_token` is silently dropped — this is fine because the forwarder never used it). Build the form's display struct from the storage struct. This is the minimum change that handles both old and new shapes.

### Pitfall 2: Wiring test fails to compile because `publisher` is a concrete type from `main.go`

**What goes wrong:** If the composition-root smoke test is placed in `cmd/pergo/main_test.go` and the test tries to call `inbound.NewDefaultRouter(chatwootSyncer, typebotForwarder)`, the test will need a `publisher` value to pass to `typebot.NewForwarder`. A real `*queue.JetStreamPublisher` requires a live NATS connection. The test may try to construct one and fail when NATS is not available in CI.

**Why it happens:** `typebot.NewForwarder` requires a non-nil `inbound.Publisher` (verified: `forwarder.go:177` calls `f.publisher.Publish` and would panic on nil pointer; the wiring-time call won't panic because Publish is not called at construction). For the *smoke test* of just the wiring (asserting that `typebotForwarder != nil`), the publisher can be `nil` — the constructor does not dereference it. The test only needs to verify the field is populated, not that the forwarder is functional.

**How to avoid:** Pass `nil` for the publisher in the smoke test. The smoke test asserts only `typebotForwarder != nil` (after `NewForwarder(...)` returns); it does not call `SyncInboundMessage`. Add a comment: "smoke test only — SyncInboundMessage path is exercised in forwarder_test.go".

**Warning signs:** If the planner tries to instantiate a real `*queue.JetStreamPublisher` in the test, the test will need a NATS container, which expands scope.

### Pitfall 3: Webhook test fails because `contact_identities` row is missing

**What goes wrong:** The handler at `typebot_webhook.go:86-91` does:
```sql
SELECT sender_identity FROM contact_identities
WHERE workspace_id = $1 AND contact_id = $2 AND channel = $3 LIMIT 1
```
If the test only calls `contactRepo.ResolveContact` (which creates a contact row but not necessarily a `contact_identities` row), this query returns no rows and the handler returns 500 "failed to resolve contact identity".

**Why it happens:** `contactRepo.ResolveContact` may not insert into `contact_identities`; it may only insert into `contacts`. The chatwoot webhook test (line 180: `contactRepo.ResolveContact(ctx, ws.ID, "telegram", "contact_tg_123", "Test Cust", "testcust", "+123")`) does not create a `contact_identities` row either, but the chatwoot handler uses a *different* resolution path (`chatwoot_mapping_repo`). The typebot handler's path is unique to the `contact_identities` table.

**How to avoid:** In the test, after `contactRepo.ResolveContact`, explicitly insert a `contact_identities` row with the test workspace, contact, channel, and a known `sender_identity` string. Use the existing pattern from any other test that exercises `contact_identities` (search codebase). Or, if `contactRepo.ResolveContact` *does* create the `contact_identities` row, document that and skip the manual insert.

**Warning signs:** Test fails with 500 + log line "failed to resolve customer identity from contact_identities" — directly indicates this pitfall.

### Pitfall 4: Form's `IsDefault` checkbox always checked

**What goes wrong:** HTML checkboxes submit `"on"` when checked and nothing when unchecked. The handler does `c.FormValue("is_default") == "true"`, but the form currently sets `value="true"` so the handler should be consistent. If the form is changed to `<input type="checkbox" name="is_default" value="true">` and the handler reads `c.FormValue("is_default") == "true"`, an unchecked box yields `"" != "true"` → `false` (correct). A checked box yields `"true" == "true"` → `true` (correct). This is fine — but the chatwoot form's existing pattern at `typebot_settings.templ:102-106` uses `value="true"` so the existing pattern is already correct.

**How to avoid:** Mirror the existing `active` checkbox pattern (lines 99-113). No new logic needed.

### Pitfall 5: Per-bot `connection_id` is stored as a `string` in `typebot.BotConfig` but a `uuid.UUID` in `connections.id`

**What goes wrong:** The form submits `connection_id` as a string. The handler must convert to `uuid.UUID` for the `contact_identities` join but store as a string in the JSON config (per the existing `BotConfig.ConnectionID string` shape at `client.go:14`). If the planner changes `BotConfig.ConnectionID` to `uuid.UUID`, every reference in `forwarder.go:68, 102` must change (`event.ConnectionID.String()` becomes `event.ConnectionID`). That expands scope.

**How to avoid:** Leave `BotConfig.ConnectionID` as `string`. The handler stores `connectionID.String()` in the JSON. No changes to `forwarder.go` or `client.go` are required.

## Open Questions

1. **Single bot vs. multi-bot UI?**
   - What we know: `typebot.BotConfig` is an array (`Bots []BotConfig`). The current form persists one. The forwarder iterates `cfg.Bots` looking for a match (`forwarder.go:67-83`).
   - What's unclear: Should the new form support *N* bots or stay single? The audit's tech-debt section does not specify.
   - Recommendation: Stay single for the form. Persist as `Bots: []BotConfig{{...}}` so the data model is ready for future multi-bot UI without another schema change. This is the minimum change that fixes the bug. (Same as the existing form's intent, expressed in line 74 of `typebot_settings.templ`.)

2. **Free-text connection_id vs. `<select>` picker?**
   - What we know: `typebot.BotConfig.ConnectionID` is a `string`. Operators can mistype a UUID.
   - What's unclear: Whether the operator console should expose a connection picker.
   - Recommendation: Use a `<select>` populated by `ConnectionRepository.List`. The form is already inside a workspace context; `ConnectionRepository.List(workspaceID)` returns the operator's connections. This is a minimal UX improvement that prevents typos. (See Discretion section in `<user_constraints>`.)

3. **Should the GET handler tolerate the old shape, or assume new shape only?**
   - What we know: The form previously wrote `{api_url, public_token, bots: [{bot_id, trigger_keyword}]}`. The new form will write `{api_url, bots: [{bot_id, public_token, connection_id, trigger_words, is_default}]}`. A workspace that stored the old shape would deserialize into `typebot.Config` with `BotConfig.PublicToken = ""` (zero value) and `BotConfig.ConnectionID = ""` (zero value).
   - What's unclear: Whether to (a) make the GET handler tolerate both shapes via a permissive intermediate struct, (b) accept data loss on re-save, (c) write a one-time data migration.
   - Recommendation: (b) Accept data loss on re-save. There are no production operators (TYPE-04 unsatisfied). Document in the plan's Concerns section. (See Pitfall 1.)

4. **Where does the composition-root smoke test live?**
   - What we know: `cmd/pergo/main.go` is `package main`; tests for it must also be `package main` (white-box) or `package main_test` (black-box, but then symbols are not accessible).
   - What's unclear: Whether to factor the wiring into a `buildInboundRouter(...)` helper function (preferred — testable) or keep it inline in `main()` and use reflection (no API change to production).
   - Recommendation: Factor the wiring into a helper `buildInboundRouter(pool, encryptor, publisher) (inbound.InboundRouter, *chatwoot.Syncer, *typebot.Forwarder, error)` and call it from both `main()` and the smoke test. This is a small refactor that improves testability. The existing chatwoot wiring at lines 212-215 moves into the helper; main() gains one line: `inboundRouter, chatwootSyncer, typebotForwarder, err := buildInboundRouter(pool, encryptor, publisher)`.
