# Phase 24.2.1: fix-typebot-message-construction-gap - Research

## Objective

Research how to implement Phase 24.2.1 to bridge PerGo's webhook payloads (specifically media attachments, fallback scenarios, and routing metadata) to Typebot's `start-chat` API, adhering to the phase's D-01 through D-09 architectural decisions.

## Findings & Gap Analysis

1. **The Gap in `StartChat`**: Currently, `TypebotForwarder.SyncInboundMessage` initializes a Typebot session without forwarding the actual text content of the inbound event. This means trigger messages or the first user message are completely dropped. 
2. **Typebot API Schema**: The Typebot `startChat` API accepts the following root properties which we need to utilize:
   - `message` (string): The initial text input provided by the user.
   - `sessionId` (string): A unique session identifier. We will override this deterministically per D-03 and D-04.
   - `prefilledVariables` (object): A key-value map for injecting variables. We'll use this for `pergo_metadata` (D-01).
3. **Session ID Determinism**: By injecting a `sessionId` as `<ConnectionID>:<SenderIdentity>`, we assure Typebot handles uniqueness for this channel and identity without collision. We still must preserve the existing `TypebotSessionID` persistence in PostgreSQL to maintain PerGo's internal state awareness.
4. **Media & Placeholders**: The inbound media from PerGo (images, etc.) must be sent as `[Media Attachment]` text placeholders (D-07), and actual media URLs must be injected into variables (D-06). Similar placeholder logic applies for unsupported payloads (locations, contacts) (D-09). 

## Actionable Plan

### 1. Update Typebot Client `StartChatRequest` Model
Modify `internal/integration/typebot/client.go` to include `Message` and `SessionID` in the `StartChatRequest` schema, and update the signature of `StartChat` to accept the entire struct (for cleaner extensibility).
```go
type StartChatRequest struct {
	Message            string            `json:"message,omitempty"`
	SessionID          string            `json:"sessionId,omitempty"`
	PrefilledVariables map[string]any    `json:"prefilledVariables,omitempty"`
}
```

### 2. Implement the Construction Logic in `TypebotForwarder`
Update `internal/integration/typebot/forwarder.go` inside `SyncInboundMessage`. Before calling `StartChat` or `ContinueChat`, compute:
- `sessionId`: `fmt.Sprintf("%s:%s", event.ConnectionID.String(), event.From)`
- `messageText`: Extract from `event.Body`, or override with `[Media Attachment]`, `[Unsupported message: location]`, etc.
- `pergo_metadata`: A flat JSON string or nested object map containing `media_url` and `media_type`.

### 3. Wire into API Calls
- **`StartChat`**: Pass the `sessionId`, `messageText`, and `prefilledVariables` (with `pergo_metadata`).
- **`ContinueChat`**: Pass the `messageText`. (Note: Typebot `continueChat` API doesn't natively take variables, so the placeholder approach allows Typebot to handle the flow, while metadata injection primarily happens at `startChat`).

### 4. Update Tests
- Update `client_test.go` to match the new `StartChat` signature.
- Update `forwarder_test.go` to assert that the `StartChat` request body contains the injected `sessionId`, `message`, and `prefilledVariables`.
